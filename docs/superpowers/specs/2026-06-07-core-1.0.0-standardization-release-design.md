# Design — `@sdcorejs/nestjs` 1.0.0: API standardization + release automation

**Date:** 2026-06-07
**Status:** Approved (design phase)
**Repo:** `sdcorejs-nestjs`

## 1. Goal

Bring `@sdcorejs/nestjs` from `0.1.6` (preview) to a stable, minimal, internally consistent
public API and publish **1.0.0** to npm. Automate publishing through Changesets on merge to
`main`. Audit all 15 entry points deeply before committing the 1.0 contract.

The README already declares the package a preview whose "API may change before v1.0.0", so this
is the sanctioned window for a clean break.

## 2. Context & constraints

- **Single known consumer:** `enterprise-platform`, which consumes the library as a *vendored
  tarball* (`file:vendor/sdcorejs-nestjs-0.1.6.tgz`), not from npm. It upgrades on its own schedule
  (planned swap `file:` → `^1.0.0`). Therefore breaking changes here are low-risk and do not block
  on downstream edits.
- **Changesets already configured:** `.changeset/config.json` (`access: public`, `baseBranch:
  main`, `commit: false`, changelog `@changesets/cli/changelog`), `@changesets/cli` devDep present.
- **Build:** `tsup` (dual ESM `.mjs` + CJS `.cjs`, `splitting: false`) + `tsc -p
  tsconfig.build.json` (emits `.d.ts` to `dist/types`). 15 entries in `tsup.config.ts` `entryMap`.
- **Existing CI/CD:** `.github/workflows/ci.yml` (push main + PR, node matrix 18.18/20:
  lint/format:check/test:coverage/build/pack dry-run/codecov) and `.github/workflows/release.yml`
  (**tag `v*`** → publish). The tag-based release conflicts with the chosen merge-to-main flow and
  will be replaced.

## 3. Decisions (locked)

| Decision | Choice |
|---|---|
| Release flow | **Changesets Release-PR** (`changesets/action` on push `main` → "Version Packages" PR → merging it publishes) |
| Audit depth | **Full deep audit** of all 15 entry points |
| Breaking posture | **Clean break** — minimal, single-canonical-path public API |
| Type-file naming | Standardize to `types.ts` (rename `context.types.ts`, `i18n.types.ts` → `types.ts`) |
| Package validation | Add **publint** + **@arethetypeswrong/cli (attw)** to the gate |
| npm secret | `NPM_TOKEN` (already used by existing workflow) |

## 4. Public API standardization rules

Principle: **one symbol, one canonical import path.** No symbol reachable from two sub-paths; no
re-export of sibling `@sdcorejs/*` packages.

### 4.1 Remove migration aliases
Drop `Sd*` aliases in [`src/orm/types/index.ts`](../../../src/orm/types/index.ts) lines 18–31:
`SdFilter`, `SdFilterOperator`, `SdOrder`, `SdPagingReq`, `SdPagingRes`. Canonical names are
`Filter` / `Operator` / `Order` / `PagingReq` / `PagingRes` re-exported (as types) from
`@sdcorejs/utils/models`.

### 4.2 Remove internal cross-module re-export duplication
`Scoped`, `TenantScoped`, `getScopedColumns` are exported from both `./orm` (origin:
`orm/decorators/tenant-scoped.decorator`) and `./tenancy` (re-export at
[`src/tenancy/index.ts`](../../../src/tenancy/index.ts) line 8). **Keep `./orm` as canonical;
remove the re-export from `./tenancy`.**

### 4.3 Keep intentional third-party re-exports (documented)
`./queue` re-exports `Processor/InjectQueue/OnWorkerEvent/OnQueueEvent` from `@nestjs/bullmq` and
`Job/Queue/Worker/JobsOptions` types from `bullmq`. This is a deliberate ergonomic gather of a
*third-party* surface (not internal duplication, not a sibling `@sdcorejs` package) — **keep**, but
add a one-line doc-comment explaining the intent. Audit other barrels for similar third-party
re-exports and apply the same rule consistently.

### 4.4 Lock down `export *` leaks
Every barrel currently uses `export *`. For each module, the audit (§5) confirms whether every
file behind `export *` is intended public API. Where a file leaks internal helpers/types, replace
`export *` with explicit named exports. Keep `export *` only where the whole file is public.

### 4.5 Type-file naming
Rename `src/context/context.types.ts` → `src/context/types.ts` and `src/i18n/i18n.types.ts` →
`src/i18n/types.ts` (and update imports + barrels). Internal-only; no public-API impact.

### 4.6 Fix `ioredis` peer dependency
`ioredis` appears in `peerDependenciesMeta` (optional) but is **missing** from
`peerDependencies`. Add `"ioredis": "^5.0.0"` to `peerDependencies` (stays optional via the
existing meta entry). It backs the redis cache backend.

### 4.7 Entry-set sync
`package.json` `exports`, `package.json` `typesVersions`, and `tsup.config.ts` `entryMap` must list
the identical set of 15 entries. Audit verifies parity (currently aligned) and the spec records the
canonical list so future additions touch all three.

## 5. Deep audit method (per module)

Modules: `index` (root), `orm`, `context`, `tenancy`, `audit`, `permission`, `cache`, `http`,
`jwt`, `validation`, `queue`, `i18n`, `action-history`, `file-storage`, `job-scheduler`.

For each, produce a findings row covering:
- **Export leaks** — internal symbols surfaced via `export *` that should be private.
- **Naming consistency** — file names, token names (`*_STRATEGY`, `*_METADATA_KEY`), class/interface conventions.
- **JSDoc** — public symbols missing or with stale doc-comments.
- **Error-code consistency** — i18n `core.*` codes thrown vs. catalog entries.
- **Dead code** — unused files/exports (cf. the already-removed `src/utils/`).

Findings aggregate into a table; fixes apply the §4 rules. Execution may be parallelized with a
review workflow (decided at implementation time).

## 6. Documentation

- **README** — change `Status: v0.1.0 — preview` to `1.0.0`; verify each of the 15 sub-paths has a
  usage section; update the peer-dependency table (ensure `ioredis` row matches §4.6).
- **CHANGELOG** — Changesets owns `CHANGELOG.md`. Remove the hand-written `[Unreleased]` section so
  `changeset version` can manage it. The 1.0.0 entry is generated from the changeset.
- **Migration notes** (`docs/migration-from-core-be.md` or a new `docs/migration-1.0.md`) — record
  the clean-break removals: `Sd*` aliases, the `./tenancy` decorator re-export, and the already-removed
  `./orm` `@sdcorejs/utils/fns` re-export, each with the canonical replacement import.

## 7. Versioning & release automation

### 7.1 Version bump
Add one **major** changeset → `changeset version` bumps `0.1.6` → `1.0.0` and writes the CHANGELOG.

### 7.2 Replace `release.yml` with Changesets action
New `.github/workflows/release.yml`:

```yaml
name: Release
on:
  push:
    branches: [main]
concurrency: ${{ github.workflow }}-${{ github.ref }}
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write       # push version commit + create GitHub release/tag
      pull-requests: write  # open/update the "Version Packages" PR
      id-token: write       # npm provenance
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: actions/setup-node@v4
        with:
          node-version: '20.x'
          registry-url: 'https://registry.npmjs.org'
          cache: npm
      - run: npm ci
      - run: npm run lint && npm run test:coverage && npm run build
      - uses: changesets/action@v1
        with:
          publish: npm run release   # = "npm run build && changeset publish"
          version: npm run changeset:version
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Flow: feature PR carrying a changeset merges to `main` → action opens/updates a **"Version
Packages"** PR (applies pending changesets, bumps version, writes CHANGELOG). Merging that PR →
action detects no pending changesets and the version is unpublished → runs `publish` → `npm
publish` (+ tag + GitHub release). No double-publish (npm rejects an existing version; the action
also guards on changeset presence).

Provenance: add `--provenance` to the publish (`changeset publish` passes npm flags via
`publishConfig`/CLI; `id-token: write` enables it). Confirm at implementation.

### 7.3 CI unchanged
`ci.yml` stays as the PR gate. Optionally add a "changeset present" check on PRs (advisory).

## 8. Verification gate (must pass before publish)

`npm run lint` · `npm run format:check` · `npm run test:coverage` · `npm run build` · `npm pack
--dry-run` · **`publint`** · **`attw --pack`**. publint + attw catch silent breakage in the
`exports`/`typesVersions` map across all 15 sub-paths (ESM/CJS/types resolution).

## 9. Out of scope

- Editing `enterprise-platform` to adopt 1.0.0 (separate task; it stays on the vendored 0.1.6 until
  it chooses to upgrade).
- `aws-sdk` v2 → v3 migration (v2 is in maintenance; flag in findings, do not change here).
- New features. This is standardization + release only.

## 10. Acceptance criteria

1. No symbol is exported from more than one sub-path (except documented third-party gathers §4.3).
2. `Sd*` aliases removed; no `export *` leaks of internal symbols.
3. `ioredis` in `peerDependencies`; `exports`/`typesVersions`/`entryMap` parity verified.
4. README status = 1.0.0 and every sub-path documented; CHANGELOG managed by Changesets.
5. `release.yml` uses `changesets/action` on push `main` with `NPM_TOKEN`; tag-based trigger gone.
6. Full gate green: lint, format, coverage, build, pack, publint, attw.
7. Version is `1.0.0` after the Version Packages PR merges; package published to npm.
